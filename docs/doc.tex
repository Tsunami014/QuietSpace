\section{Identifying and Defining}

\subsection{Problem Statement}
% Outline the problem or opportunity that the project addresses. Consider who is affected by this issue.
% Explain why this problem or opportunity is significant.
% Justify the development of a software solution as an appropriate response.
People have busy, bustling lives, and in the time between actions; when people are waiting for public transport, or for friends to arrive, or are bored, or in many other scenarioes - digital platforms have been optimised for grabbing our attention and increasing stress and anxiety on the internet. For example, many apps (especially ones paying to show above search results and on the home screen of app stores) are optimised for microtransactions through lootboxes and mindless repetition and constantly make users come back with streaks and limited time events. Additionally, the internet is a source for a lot of potentially stressful news and FOMO-inducing social media, among many other things.
This is significant as these increases of stress in an already busy life can lead to negative consequences such as burnout, anxiety and other mental health issues. Developing a space designed to be consistently interesting (through curiosity) yet still relaxing can help solve this problem. Instead of spending free moments playing high-intensity mobile games or scrolling social media, people can play this game which decreases stress while still being rewarding enough that they will play again next time and allow the game to spread to a larger audience. Ultimately, this leads to an increased mental health of users and a positive impact on society.

\subsection{Project Purpose and Boundaries}
% Outline what the project is trying to achieve.
The project will be a space on the internet for people to come to relax. It will be a game where you can move around in a world, and will be randomly generated so the world is different each time, incentivising exploration and revisiting; while containing relaxing ambience to decrease stress.

\subsection{Stakeholder Requirements}
% Identify stakeholders (client, users, teacher, peers). 
% Describe their needs, expectations, and how these influenced the project direction.
The stakeholders are the general public. Their needs are a way to fill in time or to relax that is not stress-inducing, but additionally expect the app to be engaging enough to encourage the users to replay it again. This influenced the project direction as the project couldn't just be someplace relaxing, it had to also include enough interesting features otherwise the stakeholders won't continue using the app and it will have failed.

\subsection{Functional Requirements}
% List and describe what the system must do.
The system must do the following:
\begin{itemize}
  \item The game must provide a calm environment
  \item Users must be able to make a new world, load an external world or load a world they have played before
  \begin{itemize}
    \item Users must be able to export and share their worlds in a secure and private way
  \end{itemize}
  \item Users must be able to walk around in the world
  \item Users must be able to add or remove blocks from the world
  \item Users must be able to guided on how to interact with the game
  \item The game must be able to save the world the user is in
\end{itemize}

\subsection{Non-Functional Requirements}
% List and describe system qualities such as: 
    % Performance, Usability, Security, Reliability
The system must have the following qualities:
\begin{itemize}
  \item The game must be optimised for load times
  \begin{itemize}
    \item Worlds must load within 2 seconds
    \item The game must start up within 2 seconds
    \item Throughout the game the loading of the world should be quick enough to be invisible to the user
  \end{itemize}
  \item The system must be avaliable 99.9\% of the time, ensuring avaliability
  \item The system must run reliably, logging bugs in the background to not interrupt the gameplay if possible
  \item The UI must be designed to work on various sized screens
  \item The website must be playable in any modern browser
  \item The game must store data in a secure location, protected from other users
\end{itemize}

\subsection{Constraints}
% Identify limitations affecting the project, such as: 
    % Time, Technical knowledge, Hardware/software access
The limitations on this project are:
\begin{itemize}
    \item I have not created HTML games before and have limited JS experience
    \item Procedural generation is difficult and time-consuming to get accurate
    \item I will need to curate a set of assets to ensure the game looks nice
\end{itemize}

% Analyse the functional and non-functional requirements. In your analysis, consider:
    % - Which requirements were prioritised and why

The functional and non-functional requirements were made to match the criteria outlined in the project purpose section. The requirements prioritise the usability of the app over the functionality or aesthetic, as the project is constrained by my limited knowledge of JS or procedural generation. This allows the app to be usable even if not perfect.
Other features I could have added are for example music and aesthetic. I have not made much art before and even less music, so learning art or music would most likely take longer than the time I have avaliable, and therefore I have prioritised the core functionality instead.


\section{Research and Planning}

\subsection{Development Methodology}
% Describe the development approach used (e.g. Agile, Waterfall, WAgile).
% Justify the suitability of this methodology. You could consider...
    % Project size and complexity, Time constraints, Feedback and iteration requirements
The Waterfall approach was chosen for this app due to the complexity that arises through the procedural generation. The Waterfall approach is where the development happens step by step, where the plan is laid out at the start of development and it is not changed much throughout the development.
This approach ensures the purpose, style and gameplay choices of the app remain consistant and do not overshoot the requirements laid out at the start. It is additionally helpful as the project will be plan-based due to the generation algorithm needing to be planned well, and deviations from the plan may result in creating useless assets or may make strange-looking terrain. Waterfall allows this plan to be set at the start and be followed closely.

\subsection{Tools and Technologies}
% Justify the selection of software applications, engines, developer tools, programming languages, IDEs, frameworks, libraries and/or hardware components.
% Explain how these tools supported efficient and effective development.
The project will use plain HTML and Javascript due to its simplicity and ease of use. It also allows the app to be statically hosted, and also decreases load time due to reduced dependency loading requirements. This also increases privacy and security as there is less reliance on other tools, in addition to reducing bugs from upstream changes.
That choice additionally benefits efficient development, as it makes debugging easy as browsers' built-in debuggers will use the exact code I have written instead of the compiled version of it. Lastly, plain HTML ensures compatability with all modern browsers.
